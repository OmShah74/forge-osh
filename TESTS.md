## 2. Testing Architecture & Repository Boundaries

The entire suite is structurally housed within the `/tests` directory. 

### Git Exclusion Principles
The `/tests` boundary is deliberately ignored by the git index (`.gitignore`). This actively prevents internal diagnostic artifacts, randomized temporary files generated by tool tests, and generated cache reports from leaking into the production builds. 

However, this `TESTS.md` manifesto remains actively tracked in the repository root to continuously enforce code visibility and metrics verification. 

### Verification Protocols and Reproduction Setup

To reproduce these metrics locally on a Windows machine utilizing MSYS2/MinGW, developers must follow strict isolation procedures to overcome operating system path constraints during compilation. The `dlltool` linker often struggles with absolute directory paths containing spaces (e.g., `C:\Users\OM SHAH\...`).

**Step 1: Set the Target Directory**
To bypass MSYS2 pathing errors, compilation must forcefully route to a non-spaced target directory. We utilize `C:\forge-build`.
```powershell
# Set an absolute, contiguous output path
$env:CARGO_TARGET_DIR = "C:\forge-build"
```

**Step 2: Inject the MinGW Toolchain**
Ensure the linker bindings are correctly prioritized over the standard system paths.
```powershell
# Align Cargo and MSYS64 linkers
$env:PATH = "$env:USERPROFILE\.cargo\bin;C:\msys64\mingw64\bin;$env:PATH"
```

**Step 3: Execute the Core Verification Suite**
Launch the internal test runner, ignoring standard capture protocols so logs stream actively.
```powershell
# Execute and pipe telemetry to report.txt
cargo test --test test_runner -- --nocapture | Tee-Object tests/report.txt
```

## 7. Deep Analytical Breakdown of 5 Suite Failures

During the final production evaluation phase across 227 dynamic tests, exact pass verifications reached 222 items. 5 critical items failed validation explicitly related to our persistent architecture boundaries.

### Diagnostic Matrix
1.  **`test_file_history::history_depth_increases`**
2.  **`test_file_history::snapshot_and_undo_existing_file`**
3.  **`test_tools_fs::edit_file_tool_replaces_text`**
4.  **`test_tools_tasks::task_create_and_list`**
5.  **`test_tools_tasks::todo_write_creates_file`**

### Technical Root Cause Assertions

**A. File History State Persistence Error**
The `file_history` tools inside the `forge-osh` code construct rely inherently on static global bindings to manage internal `undo` caches in `.forge-osh/snapshots`. When running under `tempfile::tempdir()`, the specific snapshot routines do not accept an injected, fully mocked `ToolContext`. As a direct result, evaluating changes generates conflicts or modifies the root runtime workspace index rather than the locked testing context.

**B. External Task Tracking Dependencies**
The `tasks` tools construct relies exclusively on an active `todo.txt` file hardcoded to map near the main executable boundary context. When mocking the environment utilizing independent asynchronous runners simultaneously creating simulated boundaries, the `task_list` function queries differing target references and subsequently fails to establish continuity resulting in read errors. Node execution attempts to verify created tasks, but finds the mock directory empty.

**C. Line-Endings Parsing (`edit_file_tool`)**
Running complex line modifications relies strictly upon newline configurations inside mock parameters. Tests simulating exact replacement vectors occasionally invoke Unix-style specific `\n` parameters against raw Windows `\r\n` files dynamically generated by `std::fs::write()`. The diff analyzer fails to confirm positive replacements during character-to-character validation arrays resulting in minor compilation failures across tests heavily bound to string evaluations.

### Immediate Reengineering Roadmap

Based off these documented `TESTS.md` assertions, development velocity should prioritize:
1.  **Refactoring the Context Boundary:** `file_history` algorithms must natively embrace `ToolContext` configurations directly in execution vectors rather than implicit project root environments to allow pure mock abstractions during development.
2.  **File Normalization Buffers:** `fs::edit_file` deployment routines should algorithmically strip and normalize trailing carriage returns dynamically prior to executing the `similar` diff engine libraries. This removes reliance natively bound to operating framework idiosyncrasies.
